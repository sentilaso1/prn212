	II. Requirement Specification
II.1 Problem description
WorkFlowPro is a web-based team task management platform built to replace fragmented collaboration through chat messages and spreadsheets. The system is designed for small and medium teams that need one shared workspace to create projects, assign tasks, coordinate execution, monitor status in real time, review outcomes, and maintain a transparent activity history. Compared with manual task tracking, WorkFlowPro centralizes task ownership, workflow states, member roles, notifications, and KPI-oriented reporting so that project managers and members can collaborate more consistently.
II.2 Major Features 
1. FE-01: User Authentication and Account Security: Allows users (Guest, Member, Project Manager) to register accounts, verify via OTP, sign in with Email, sign out, reset passwords, and manage account security. Admin accounts are provisioned directly via the system database.
2. FE-02:  Task Evaluation and Performance Scoring System: Allows Admins to evaluate completed tasks by providing scores and feedback. The system automatically calculates performance scores for Members based on task completion, deadlines, and evaluation results.
3. FE-03:  Performance Dashboard and Analytics: Provides Admins and Project Managers with performance dashboards, including evaluation scores, historical performance trends, and user levels (Junior, Mid, Senior) to support level adjustment decisions.
4. FE-04:User Level Adjustment Request Workflow (PM → Admin):Allows Project Managers to submit requests to adjust user levels (e.g., promotion or demotion such as Junior, Mid-level, Senior) based on performance data. All requests must be reviewed and approved by Admins before taking effect. The system does not support role changes, only level adjustments within a role.
5. FE-05: User Profile and Performance Management (Admin):Allows Admins to view and manage user profiles, including roles (fixed), levels (Junior, Mid, Senior)

	Summary:
	Allows a Guest to create a new account by selecting an account type (Member or PM). Member accounts are verified via OTP and activated immediately. PM accounts skip OTP and wait for Admin approval before gaining access.0
	Preconditions:
	The user does not have an existing account.


	Postconditions:
	(Member) Account is active; user is logged in and can join a Workspace. (PM) Account is in Pending Approval state; user can only view the waiting page until Admin approves (UC-13).
	Dependency: 
	UC-13 (Admin approves PM registration)
	Description of main sequence:
	Description of main sequence (Path A — Register as Member):
1. Guest navigates to /register.
2. Guest selects account type: Member.
3. Guest fills in: email (must be @gmail.com domain), display name (optional), password (≥8 characters, at least 1 uppercase + 1 number), confirm password.
4. Guest clicks "Create account".
5. System validates input, then sends a 6-digit OTP to the provided email (valid for 10 minutes).
6. Guest enters the OTP on the verification screen.
7. System verifies OTP → activates account with role Member.
8. System automatically logs the user in and redirects to the Workspace joining page.
Description of main sequence (Path B — Register as PM):
1. Guest navigates to /register.
2. Guest selects account type: PM — workspace will be created after Admin approval.
3. Guest fills in: email (any domain accepted), organization / company name (required — becomes the Workspace name upon approval), display name (optional), password (≥8 characters, at least 1 uppercase + 1 number), confirm password.
4. Guest clicks "Create account".
5. System validates input → creates account with status Pending Approval and notifies Admin of a new PM request (UC-13).
6. User is redirected to a waiting page: "Your PM account request is under review. You will be notified via email once approved."
	Description of alternative sequences:
	Step 3 (Path A): Email is not a @gmail.com address → display: "Member accounts require a Gmail address (@gmail.com)."
Step 3 (Path A or B): Email already registered → display: "This email is already registered. Please sign in or use a different email."
Step 3 (Path A or B): Password and confirm password do not match → display: "Passwords do not match."
Step 6 (Path A): OTP entered is incorrect → display: "Invalid OTP. Please try again." Allow up to 3 attempts.
Step 6 (Path A): OTP expired → display: "OTP has expired." Provide "Resend OTP" option; new OTP invalidates the previous one.
Step 6 (Path A): After 3 failed OTP attempts → session terminated; user must restart from Step 1.


	Exceptions:
	Step 5 (Path A): Email Delivery Service fails → display: "Unable to send OTP. Please try again later."
	Nonfunctional requirements:
	* Password must be hashed (bcrypt) before storing in the database.
* OTP is 6 digits, valid for 10 minutes, single-use, invalidated upon resend.

	08_Evaluate Task


UC ID and Name:
	8. Evaluate Task
	Created By:
	Baonvg
	Date Created:
	14/01/2026
	Primary Actor:
	PM
	Secondary Actors:
	None
	Summary:
	After a task reaches Done, PM scores the task (1–10) and optionally proposes a level change for the assigned Member. The evaluation is open for dispute by other PMs in the Workspace within 24 hours. After 24 hours the evaluation is locked and scores are finalized.
	Preconditions:
	* Task has been moved to the Done column (UC-06).
	Postconditions:
	* TaskEvaluation record saved; Average Score and Completion Rate updated after the 24h window closes; Member notified of final result.
	Dependency: 
	UC-06 (triggers evaluation), UC-10 (level change proposal), UC-12 (notification)
	Description of main sequence:
	Task moved to Done → PM receives realtime notification: "Task [Name] needs evaluation."
PM opens the evaluation modal and enters:
* Score (1–10, required)
* Comment / feedback (optional)
System saves the evaluation with status Open and starts a 24-hour dispute window.
All other PMs in the Workspace receive a notification: "Evaluation for task [Name] is open for review (24h)."
During the 24h window, any other PM can open the evaluation and submit a dispute (see alternative flow).
After 24 hours, the evaluation window closes automatically:
* If no dispute: evaluation is locked; scores finalized.
* If dispute exists: the disputing PM's comment is attached; the original PM is notified and may revise their score once before final lock.
System calculates and updates:
* Completion Rate = Tasks Done on time / Total tasks assigned
* Average Score = Mean of all evaluated task scores
System saves finalized evaluation to TaskEvaluations and updates MemberProfile.
Member receives realtime notification with their final score and feedback.
(Optional) If PM believes a level change is warranted → PM submits a Level Adjustment Proposal (UC-10).
	Description of alternative sequences:
	Another PM disputes the evaluation (within 24h):

	Exceptions:

	Nonfunctional requirements:
